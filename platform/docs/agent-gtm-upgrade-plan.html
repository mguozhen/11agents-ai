agent-gtm · 升级计划 · 2026-05-18

从 playbook 生成器
升级成闭环增长 OS

触发源:用户分享的 Gins AI pitch deck —— 一套 "self-learning Growth Operating System"。 Gins 的内容不能抄(它优化的是旧漏斗),但它的形式值得偷。 本计划把 agent-gtm 从「跑一次出一个静态 deck」升级成一个在 6 个产品上持续运行、可测量、会学习的系统。

01 — 分歧

Gins 和 agent-gtm 在赌不同的未来

先把定位钉死。两者不是竞品,是两条岔路 —— 抄错了就变成「给 agent 做的 Gins」,失去全部差异化。

Gins AI

把旧的增长 playbook 跑快 1000 倍

synthetic persona、A/B、campaign personalization、CRM / BI / ad channel 集成。

客户依然是人类营销团队;优化的是「人怎么被触达」。

优化 — 你怎么够到人

agent-gtm

playbook 本身正在被替换

MCP catalog、agent router metadata、OpenAPI cookbook、hub-and-spoke cross-link。

产品的新购买入口是 AI agent;优化的是「agent 怎么找到你」。

优化 — agent 怎么够到你

一句话写进 showcase 总览页: Gins 优化你怎么够到人;agent-gtm 优化 agent 怎么够到你。不同的漏斗。

02 — 偷形式,不偷内容

Gins deck 做对、agent-gtm 现在缺的 3 件事

Gins slide 6
OS,不是文档

Gins 是在跑的闭环 Understand → Optimize → Execute。

agent-gtm 跑一次出一个静态 HTML。6 产品 = 6 个一次性 deck,不是 1 个 OS 的 6 个实例。

Gins slide 5
量化的痛

每个动作有 Delay / FTE 成本 / 战略影响 / $ 损失,年损 $4–7M。

我们有 done/partial/todo 进度条,但没说「不做的代价」。

Gins slide 5 角标
Why now

91% marketer 用 AI、CMO 预算转移、first-mover 窗口。

我们的 showcase 没有这个时间钩子。

03 — 升级 A

从 playbook 生成器 → 闭环 OS

A · 骨架

让 audit 状态可重跑、有历史、能 diff

最大结构升级 —— B 和 C 都把数据写进 A 定义的新 schema。
现状
shipped[] 里 14 个 item 的 done/partial/todo 状态每次手打进 JSON。 重跑 = 从头重写。没有上一次的快照,看不出本周 vs 上周的速度。
升级后
playbook-data/<slug>.json 成为唯一状态源。20 项 audit 结构化: 每项带 status / note / last_checked,分数自动算。重跑 = re-audit + diff 上次快照。
新增 references/audit-items.json — 把 20 项 checklist 变机器可读(layer / id / check 命令)
新增 scripts/audit.py — 读 audit-items + 当前 state,辅助重新打分,写回 JSON
新增 playbook-data/history/<slug>-<date>.json — 每次 audit 留快照
新增 references/loop.md — Understand→Optimize→Execute 循环说明
改 SKILL.md — 从 1 个动作拆成 3 个子命令:audit / moves / render
改 render.py — 加 diff 视图:本次 vs 上次快照,标出新 done 和回退项
交付物 — agent-gtm audit btcmind 重跑后输出 「audit 10/20 → 12/20,本周新 done 2 项,0 回退」,而不是重写整个 deck。
04 — 升级 B

Agent 发现模拟 —— 我们版的 Persona Bank

B · 测量层

真起一个 buyer-agent,测它能不能找到产品

Gins 模拟合成的人类 persona;我们模拟真实的 agent 发现 —— 可测、可重复、不是合成数据。 btcmind.json 的 W7-8 已埋「Persona Bank V1 / Persona Lab / v3 plan §B」占位,这一步把它接上。
现状
audit 表全是「我们做了什么」的自评。没有任何一条来自外部信号 —— 没人知道 agent 实际能不能在工具索引里看到产品。闭环缺测量层。
升级后
给一个真 agent buyer-task(「找个分析 Amazon 评论的工具」), 看它是否经 MCP catalog / 搜索 / 工具索引命中产品 —— 命中没?第几位?走哪条通道?
新增 scripts/discovery_sim.py — 走 flatkey OpenAI 代理跑 buyer-agent,判定命中 / 排名 / 通道
新增 references/discovery-sim.md — 每个品类的 buyer-task 模板(电商 / 加密 / CS / 支付…)
改 playbook-data/<slug>.json — 加 discovery_sim 字段:task / found / rank / channel / excerpt
改 render.py — 在 deck 渲染「发现模拟」结果块,作为 Card 1 的真实 KPI
交付物 — 每个产品一条客观分: 「buyer-agent 找 Amazon 评论工具 → 第 4 位命中 voc,经 mcp.so」。闭环从此有真实输入信号。
05 — 升级 C

量化痛 + 杠杆表

C · 叙事

给每个产品页加「不做的代价」和 1000x 杠杆

最便宜的一项 —— 主要是 render.py + 模板 + 填数据,不动 skill 架构。可与 B 并行。
现状
deck 说了「该做什么」,没说「不做会怎样」。 也没把 human-team 时间 vs CC 时间的压缩显示出来 —— Gins 的「1/1000x velocity」我们其实更有资格讲。
升级后
每产品页一张 pain table(GTM 动作 / 现在手工怎么做 / 不做的代价 / agent 时代打法)+ why-now 钩子 + 每个 move 标 human / CC 双尺度耗时。
改 render.py — 渲染 pain_table / why_now / move 上的 effort
改 references/playbook-template.html — pain table + why-now 区块的 CSS / 结构
改 references/four-cards-rubric.md — 记录新字段写法
改 playbook-data/*.json ×6 — 补 pain_table / why_now / 每个 move 的 effort
交付物 — 每张 deck 顶部一张量化痛表,把「为什么现在必须做」讲成数字。
06 — 落地顺序

分 5 个 Phase,A 打地基,B+C 并行

B 和 C 都写进 A 定义的新 JSON schema,所以 schema 必须先定。耗时按 CC 估,括号里是人类团队等价工作量。

  1. Phase 0 · 定 schema(地基)
    重构 playbook-data JSON 结构:结构化 audit[] 20 项 + 预留 discovery_sim / pain_table / why_now 槽位;写 audit-items.json。
    CC ~30 min(human ~半天) · 无依赖,先做
  2. Phase 1 · A 闭环化
    SKILL.md 三子命令 + audit.py + history 快照 + render.py diff 视图 + loop.md。
    CC ~45 min(human ~1.5 天) · 依赖 P0
  3. Phase 2 · B 发现模拟
    discovery_sim.py 走 flatkey 代理 + buyer-task 模板;先在 voc + btcmind 上验证。
    CC ~1 h(human ~2 天) · 依赖 P0 · 可与 P3 并行
  4. Phase 3 · C 量化痛表
    render.py + template 加 pain table / why-now / 双尺度 effort;补 6 个产品数据。
    CC ~1 h(human ~2 天) · 依赖 P0 · 可与 P2 并行
  5. Phase 4 · 全量重跑 + 重新部署
    6 个产品全部过一遍新 OS(audit→moves→render),刷新 agent-gtm-showcase.html,重部署 Vercel。
    CC ~40 min(human ~1 天) · 依赖 P1–P3

关键收益 — 做完之后,「用 6 产品锻造 1 skill」这句话第一次成立:6 个产品不再是 6 个一次性 deck,而是同一个 OS 的 6 个持续运行实例,每周能看到 audit 分数和发现率的变化曲线。
