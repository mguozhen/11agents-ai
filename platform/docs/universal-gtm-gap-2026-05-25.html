§ 01 · PRINCIPLE vs REALITY

原则 vs 实装 —— 差在哪

P04 说

  • 卖什么都行 —— swarm 是 capability,不绑特定产品
  • traffic + revenue 是唯一 KPI
  • 渠道随产品需要而变,不是固定的
  • swarm adapts to whatever you need to sell
→ vs →

2026-05-25 实际

  • 11 个 agent 写死在 04-content-strategy.md 的 hydration YAML schema
  • 0 处涉及 conversion / utm / revenue(grep 实证)
  • engines/<slug>/ 是手写 audience/voice/platforms,新产品要手动建 30+ 文件
  • Iron Triangle Reviewer 是每产品每 agent 一个真人 → 不 scale

原则在墙上,执行没跟上。差距不是原则要改,是平台要补。把这 4 处补上,就兑现了 P04 的承诺。

§ 03.1 · INVARIANT 1
I1 · 85% ✅

任何垂直都能 onboard

「输入一个 url 或产品描述,4-step ContentOS 跑得通,无论产品是律所、SaaS API、还是 NFT collection。」
现状
Step 1 (Market Insight) 已 universal —— CIA 方法论,5-8 tracks,任何垂直都适用。

但:engines/<slug>/ 下的 audience / voice / platforms 是手写的 30+ markdown 文件。 目前只有 voc-ai 完整,solvea 半完整。新产品 onboard 后这一层是空的。
universal 标准
Step 1 输入产品 → Step 4 输出后,系统自动生成完整 engines/<slug>/:audience 2-4 segment / voice 1 brand-voice / platforms 8 个 platform-specific tone。

人不需要手动写一行 md。
要 build
① Engine Hydrator:接 Step 4 输出,LLM 自动生成 engines/<slug>/ 全套.
② Template Library:把 voc-ai 的 engines 抽成 vertical 模板(B2B SaaS / DTC / dev-tool / crypto / service 5 套).
③ Quality gate:hydrator 跑完后 lint:每个文件 ≥ 200 字、有具体例子、与 brand-voice 一致。
§ 03.2 · INVARIANT 2
I2 · 20% ❌

任何渠道都能插拔

「想加 Producthunt / HN / TikTok Shop / Discord / Substack / Apple Search Ads —— 不改 schema、不改 prompt、不动 hydrate-agents.py。」
现状
11 个 agent 硬编码在 3 个地方:
· ROSTER.md(11 行表)
· templates/04-content-strategy.md 的 YAML schema(01-foundation...11-poster)
· scripts/hydrate-agents.py(parser 写死)

想加第 12 个 agent → 3 处都要改 + 重跑所有产品的 Step 4。
universal 标准
Agent Registry 表(agent_definitions):每个 agent 是一行 db 记录,有 name / platforms / kpi_schema / builder_skill / reviewer_template。

Step 4 prompt 改成「读 registry 选 N 个 agent activate」—— prompt 不写死名字。

想加 PH agent?往 registry insert 一行就行。
要 build
① 新表 agent_definitions + workspace_agents(per-product 启用关系).
② Migration:把现 11 agent 灌进 registry.
③ Step 4 prompt 改写成动态版.
④ Hydrator 改成 schema-driven.
§ 03.3 · INVARIANT 3
I3 · 10% ❌

Reviewer 可扩展

「100 个产品 × 11 agent = 1100 个 Iron Triangle 槽位。不能塞 1100 个人。」
现状
P03 立法「No Triangle = No Agent」 —— 每个 agent 需要 1 个真人 Reviewer。

6 个产品 × 6 active agent ≈ 36 槽位,目前已经塞不下(5 个 TBD slot 卡 P0 招人)。

往 ∞ product 走完全不可能。
universal 标准
Reviewer 分 3 层:
· L1 模板 reviewer — content 通过 rubric 自动过(80% 流量)
· L2 样本 reviewer — 真人审 10% 抽样,刷新 rubric
· L3 升级 reviewer — 异常 / 高价 content 真人逐条审

1 个真人可以 cover 10 产品 × 11 agent.
要 build
① Rubric system:每 agent 有 review_rubric.json,5-10 条可机器判规则.
② Auto-grader:接 LLM-as-Judge,给每 draft 评 0-100.
③ Sampling cron:每周抽 10% 给真人 reviewer,反写 rubric.
④ Escalation:>10% draft 不过 / 单 content 预期曝光 > 阈值 → L3 真人.
§ 03.4 · INVARIANT 4
I4 · 0% ❌

转化闭环可度量

「P04 说 traffic + revenue is the goal —— 但今天平台 0 行代码涉及 conversion / utm / revenue。grep 实证。」
现状
Pipeline 在 Published 就断了。Bank tab 有 7 条 published,但没有任何字段记录「哪条带来了多少 signup / 多少 revenue」。

grep -rE "conversion|utm|attribution" 在 app 代码里 0 命中。

→ 没有反馈 → swarm 无法 learn → 第 100 条 content 不会比第 1 条更好。
universal 标准
每条 published 出口自带 tracking handle(UTM + 短链 + post-id),后端跑 定时归因 job:

· 拉 GA4 / Plausible / 自有 db 看每条 link 的 conversion
· 写回 conversion_events 表(content_id, channel, signups, revenue_usd, ts)
· Dashboard 出 per-agent ROI 看板:每个 agent 这周给每个产品贡献了多少 traffic+revenue
要 build
① Schema:conversion_events + tracking_handles 两张表.
② 出口注入:agent runner 在发布时自动生成 UTM/短链/post-id 写入 tracking_handles.
③ Attribution cron:每天拉 GA4 / 自有 db,merge 回 conversion_events.
④ ROI dashboard:Foundation tab 新增 "Per-Agent ROI" 卡.
⑤ Feedback loop:rubric grader 拿 ROI 信号调权.
§ 03.5 · INVARIANT 5
I5 · 50% ⚠

Mix 自动决策

「Step 4 prompt 已经会 activate / skip channel,但只在 11 个固定 agent 里挑。I2 解锁后 I5 才能真正 universal。」
现状
Step 4 LLM 会判断:B2B SaaS → skip 11-poster;crypto → lean reddit/social。这个已经在 prompt 立法.

但 channel pool 还是 11 个写死的.LLM 没法说「这个产品要加个 PH 上线 agent」.
universal 标准
Step 4 输入 = 产品 + agent registry(N 个).
输出 = (active subset, per-agent KPI, expected ROI rank).

人在 dashboard 上可以接受 / 否决 / 调整,但 default 是 swarm 自己决定.
要 build
① 等 I2 解锁 registry.
② Step 4 prompt 改造:不再列 01-11,改成「从 registry 里挑」.
③ 加 ROI 先验(I4 落地后):告诉 Step 4 prompt「同类产品 channel X 历史 ROI 12.3」.

§ 05 · LITMUS TEST

universal 试金石 —— 3 个跟现有 6 个完全不同的产品

系统是不是真 universal,不靠自我评估,靠能否独立 onboard 这 3 个跟现有 6 产品无亲缘的样本。 都跑通 + 都能在 30 天内产出有归因数据的 traffic+revenue → 通过.

Sample A · Service business

邻家律所 / 一家社区律师事务所

完全不是 SaaS / 不是 Amazon / 不是技术产品。本地 service。
价位 $200–$2000/case · 受众小且地域化.
  • I1 · engines/lawyer-firm/ 自动 hydrate
  • I2 · 自动启用 Yelp / Google Maps / 本地 SEO,跳过 reddit
  • I5 · auto-mix 给出 3 channel 重点
  • I4 · 4 周后能看到 X 个 inbound consult
通过条件:30 天 ≥ 3 个 inbound + UTM 归因
Sample B · Hardware

智能宠物喂食器(Kickstarter)

实体硬件 · 一次性 $99–$199 销售 · 受众 DTC.
跟所有现 6 产品形态完全不同.
  • I1 · 4-step 跑出 hardware-vertical 风格的 brief
  • I2 · 自动启用 IG / TikTok / Kickstarter / 宠物 reddit
  • I5 · 跳过 SEO blog · 重 UGC + KOL
  • I4 · KS 页面 traffic + 真实 pre-order 归因
通过条件:30 天 ≥ 10 个 pre-order 可溯源
Sample C · Agent-native

纯 MCP 工具(没有 UI)

P06 立法的目标 —— agent 是新分发渠道.
产品没有人类 UI,只暴露 MCP / OpenAPI.
  • I1 · 4-step 不假设产品有官网
  • I2 · 自动启用 mcp.so / smithery / awesome-mcp / agent 编辑器集成
  • I5 · 完全跳过传统社交,重 agent registry / SDK demo
  • I4 · MCP tool call 计数 / API key 注册数 归因
通过条件:30 天 ≥ 50 次 MCP 调用 来自 ≥ 5 个 agent host
