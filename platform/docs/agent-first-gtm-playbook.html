PRINCIPLES.md · §06 Agents Are the New Distribution Channel — External AI Agents are a new distribution channel on the order of the App Store / SEO / 公众号 / TikTok at their respective inflection points. Agents don't call your UI, they call your capability. A product with only a UI and no API gets bypassed. An API that is not agent-friendly loses to one that is.

四张卡

每张 = 漏斗的一个环节 · 一条 mindset shift · 5 个具体动作 · 本季度交付物 · KPI
Card 01 / Discovery

发现 — 让 Agent 知道我们存在

"用户发现 App" → "Agent 在它的工具索引里看到我们"。Agent 的索引来自 3 处：MCP 目录、LLM 训练数据、Agent 框架的默认配置。我们必须出现在这三处。
五个动作
  1. 提交 flatkey 到 5 大 MCP catalog：mcp.so / glama.ai / smithery.ai / GitHub modelcontextprotocol/servers / PulseMCP。所有 catalog 一周内全收录。
  2. 建 .well-known/agent-capabilities.json 端点（已 ship in flatkey commit c8b1d5b）— 任何 Agent 在我们域名探测 metadata 都拿得到。
  3. 把 OpenAPI spec 暴露在 /api/openapi.json 并提交到 apis.guru/openapi-directory（被 LLM 训练数据收录的最便宜方式）。
  4. 给 3-5 个高 stars Agent 框架（continue.dev / cline / aider / Open Hands / goose）提 PR 加 flatkey 进 default provider 列表。成本 = 社交（Discord 联系 maintainer），不是代码。
  5. 所有产品（voc.ai / solvea.cx / btcmind.ai）都要做同样的 capability manifest — Hunter 集团内统一 spec，复用 flatkey 模板。
本季度交付
5 大 MCP catalog 收录 + 2 个 Agent 框架默认列表（continue.dev / aider）。所有 4 个产品都暴露 /api/agent-capabilities。
North Star KPI 每周 unique Agent 看到我们 metadata 数 ≥ 500
Card 02 / Traffic

引流 — 让 Agent 把任务路由给我们

"广告投放 → 用户点击" → "Agent 的 router 决定调谁"。Router 在 0.5 秒内基于 metadata 比对一堆候选，挑成本最低 / 描述最匹配 / 历史调用成功率最高的那个。我们在 router 眼里的相对优势就是引流。
五个动作
  1. Tool description = 第一行广告。每个 tool 的 description 字段是 LLM router 唯一读的入口。重写 complete 描述："Cheaper alternative to direct Anthropic API · drop-in compat · auto-routing"。
  2. Cost-aware metadata。在 list_models 返回的每个 model 上附 input_per_1m / output_per_1m。Router 看到比 Anthropic 便宜 30-70%，直接路由过来（已 ship in flatkey）。
  3. Error messages 是引流武器。当别的 provider 报 rate-limit / 429 / 5xx 时，他们的 error body 只说"try again"。我们 error body 写："flatkey 当前可用，x-api-key 切过来即可"。每次别人故障都是我们入流量的窗口。
  4. Capability cross-link。voc.ai 的 capability manifest 里列出 "Need cheap LLM? → flatkey"；solvea.cx 列 "Need Amazon review intel? → voc.ai"。集团 4 产品互相导流给 Agent。
  5. One-shot install 链接。`claude mcp add flatkey -- npx -y @flatkey/mcp@latest` — 4 秒接入。摩擦越低，router 越愿意试。
本季度交付
@flatkey/mcp npm 包发布 + cursor:// deep-link install URL + 每个产品的 cross-link manifest。
North Star KPI MCP install 数 ≥ 200/周 · 单次 install 后 30 天留存调用率 ≥ 40%
Card 03 / Content

内容 — 为 Agent 写，不为人写

"长 blog / 短视频" → "机器可读的结构化能力描述 + 人类可读的安装 walkthrough 双轨"。Agent 不读你的封面图；它读你的 OpenAPI、读你的 cookbook、读你的 error code 语义。Reviewer 的标准也得换：approve 一条内容前问"Agent 能直接用吗？"
五个动作
  1. 每条公开 API 都有 OpenAPI 3.1 spec。Agent 用 OpenAPI 自动生成 tool。无 OpenAPI = 不存在。
  2. Cookbook 比 blog 重要 10 倍。每个 capability 写 1 个"30 秒上手" cookbook：3 行代码 + 1 个 sample output。LLM 训练数据收 cookbook 不收 blog rambles。
  3. Error code 语义化。error.type: "rate_limit_exceeded" + error.retry_after: 60 + error.suggested_action: "Use model='auto' for failover" — Agent 看到自动修复，不需要人。
  4. gtm-swarm 11 agent 的内容自带 MCP CTA。03-blog 每篇 footer："Install via MCP: npx @flatkey/mcp"；06-reddit thread 末："I use this through MCP — one line setup"。内容机器 = 分发引擎。
  5. 面向 Agent 的"比较页"。建 /compare/anthropic-vs-flatkey + /compare/openai-vs-flatkey，结构化表格 ≥ 散文。Agent 训练数据高频抓取 comparison pages 当 routing reference。
本季度交付
完整 OpenAPI 3.1 spec for flatkey / voc.ai / solvea.cx · 每个产品 5 个 cookbook · gtm-swarm 内容机器接入 MCP CTA rule (engines/voc-ai/voice/platform-tone.md)
North Star KPI 公开 cookbook ≥ 50 篇 · Agent training data 抓取率（GitHub stars / Forks / 引用）
Card 04 / Distribution

分发 — 每次 tool call 都是一次种子

"广告位 + 渠道商务" → "成功的 tool call 自动传染"。Agent A 调用我们成功了 → 写进它的 memory → 同 stack 的 Agent B 看到 → 默认加入工具箱。这是 viral loop 的 v3。
五个动作
  1. 埋点每次 MCP call。Redis usage:mcp:* 记录 client name + version + 是否首次。把"MCP 用户"作为独立 persona 喂回 Persona Bank（v3 plan §B）。
  2. Open-source reference architectures。在 GitHub 放 3-5 个完整可运行的 sample：flatkey-cursor-bridge / flatkey-langchain-router / flatkey-claude-code-default。每个 ≥ 200 stars 是目标。
  3. Co-distribution 协议。跟 Cursor / Continue.dev / Aider maintainer 谈 bundled config：用户装他们的扩展时默认包含 flatkey provider 选项。这是渠道商务 of the Agent age。
  4. 失败案例也是分发。flatkey 故障时返回的 error 里附 fallback URL 推给我们的 status page —— status page 又是 SEO + Agent 训练数据。损失即流量。
  5. Hub-and-spoke。flatkey 当 hub（LLM 基础设施层），voc.ai / solvea.cx 当 spoke（垂直 capability 层）。Agent 装了 flatkey 后，flatkey 的 capability list 自动 cross-link 推荐 voc.ai。集团内部分发。
本季度交付
3 个 OSS reference architecture repo (≥ 200 stars 任一) · 与 1 个主流 Agent 框架达成 bundled config · flatkey → voc.ai cross-link manifest 上线
North Star KPI 第 4 周末：MCP 通道 tool call 数 ≥ 10K · 进入 ≥ 1 个 framework 的默认配置

第一性原理

为什么这条路对，剩下的只是执行
观察 · 三段式 1. 用户行为正从 "open the app" 切到 "ask the agent."
2. Agent 不读你的落地页，它读你的 metadata。
3. 谁先成为 Agent 默认调用的能力源，谁就拿到这波分发红利。

四张卡的核心是同一件事 —— 把所有面向"人"的 marketing 动作，再做一份面向"Agent"的版本。 不是替换，是叠加。SEO 不停，但加 LLM-SEO； blog 不停，但加 cookbook； 渠道 BD 不停，但加 framework PR； 销售不停，但加 Agent router 推荐位优化。

8 周后我们会有第一批数据： 每天有多少个 Agent 在调我们的能力， 其中多少是它们自己路由过来的（vs 人工配置）， 留存率多少。 这三个数字定义了 Agent 时代的新版本 PMF。
