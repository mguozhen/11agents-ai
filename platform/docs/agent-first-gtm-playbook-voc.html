四张卡

Card 01 / Discovery

发现 — 让 Agent 知道这条 capability 存在

"Amazon 卖家 SEO 找你" → "Agent 在它的工具索引里看到你"。索引来自 3 处：MCP catalog、LLM 训练数据里抓到的 GitHub repo、Amazon-垂直 Agent 框架的默认配置。垂直产品的分类不能跟 flatkey 抢 "AI / LLM" 赛道，要进 "Ecommerce / Research"。
五个动作
  1. 5 大 MCP catalog 立刻提交 — mcp.so / glama.ai / smithery.ai / modelcontextprotocol/servers GitHub PR / PulseMCP。分类选 Ecommerce / Research，不要选 AI。
  2. GitHub topics 立刻配齐：mcp mcp-server model-context-protocol amazon-fba amazon-seller-tools voc review-analysis helium10-alternative。Anthropic / GitHub 的 MCP 发现页直接按 topic 检索。
  3. OpenAPI 3.1 spec for MCP tool layer —— 不是 Shulex 那层，是我们 4 个 tool 的层。放 /api/openapi.json 供 Agent 自动生成 wrapper。
  4. 加 .well-known/ai-plugin.json + .well-known/agent-capabilities.json —— 没自有域名，挂 GitHub Pages：mguozhen.github.io/voc-amazon-reviews/.well-known/...
  5. Cline + Open Hands + Cursor 的 "Amazon FBA preset" 提 PR 加 voc-amazon-reviews MCP。**比直接进 default LLM provider 容易得多**，因为这些 framework 已经在收 use-case-specific preset。
本季度交付
5 大 MCP catalog 收录 + 自有 OpenAPI spec for MCP layer + 2 个 Agent framework 的 Amazon preset PR merged。
North Star KPI MCP catalog 累计 install 数 ≥ 200 · GitHub stars 从 16 → ≥ 100
Card 02 / Traffic

引流 — 让 Agent 路由 Amazon 任务给我们

"广告投放 → 卖家点击" → "Agent router 决定调谁"。Amazon 评论分析 task，候选有 Data Dive MCP / 自研爬虫 / Helium 10 API。我们要在 description / metadata / cost 三个字段上比对手清晰。垂直产品的 hook 不是 "便宜 30%"（flatkey 那套），而是 "Helium 10 替代品 / 开源 / 免费试用"。
五个动作
  1. 重写 4 个 tool 的 description 字段 让 LLM router 一眼读懂：从 "Fetch raw Amazon reviews via Shulex VOC API" 改为 "Get verified Amazon reviews (10 marketplaces, up to 1000/ASIN, real OpenAPI not scraper) — Helium 10 alternative, free starter tier"。
  2. Cost-aware metadata：在 tool schema 加 _meta.credits_per_call（100 条评论 ≈ 60 credits ≈ $0.05）。Agent router 看到便宜的 capability 优先调。
  3. Error message 当广告位：当 ASIN 无效 / Shulex rate-limit 时，error body 写 "Need a key? Free signup: apps.voc.ai/openapi?utm_source=mcp_error"。别家 5xx 时也跳过来。
  4. One-shot install 必须 4 秒：发 @voc-amazon-reviews/mcp npm 包，claude mcp add voc-amazon-reviews -- npx -y @voc-amazon-reviews/mcp 一行能用。现在的 clone-repo + pip-install 是流量杀手。
  5. Cross-link from flatkey & solvea：flatkey 的 capability manifest 列 "Need Amazon review intel? → voc-amazon-reviews"；solvea.cx 同。Hub-and-spoke 集团内部分发。
本季度交付
@voc-amazon-reviews/mcp npm 发布 + 4 个 tool description 重写 + flatkey → voc cross-link manifest 上线。
North Star KPI MCP install ≥ 100/周 · 单次 install 后 30 天留存调用率 ≥ 35%
Card 03 / Content

内容 — 为 Amazon-seller Agent 写，不为人写

"长 blog / 短视频" → "机器可读的 cookbook + 结构化的 comparison page"。人读 README 找 "是不是骗子"；Agent 读 cookbook 找 "3 行代码能不能 demo"。我们的内容机器还在写给人，要双轨。
五个动作
  1. 5 个 30 秒 cookbook：(1) "Analyze any ASIN in 3 lines" (2) "Compare your listing vs top competitor" (3) "Daily review monitor cron" (4) "Generate listing copy from VoC" (5) "Pull reviews into your own DB"。每个 ≤ 30 行 markdown。
  2. Error code 全部 semantic 化：error.type + error.suggested_action。例：type="voc_api_no_key" / suggested_action="Visit apps.voc.ai/openapi to get a free starter key, then export VOC_API_KEY"。Agent 读到 suggested_action 自动修复，不用问人。
  3. Comparison page 单独建 — mguozhen.github.io/voc-amazon-reviews/compare/data-dive + /compare/helium10 + /compare/junglescout。结构化对比表，不是散文 blog。LLM 训练数据高频抓 comparison pages。
  4. README 顶部加 "Quick install for Agents" 段：给 Cursor / Claude Code / Continue.dev 各贴一段 install 配置；README 是给人读的 "为什么是 moat"，再加一层机器读的 quick-install。
  5. gtm-swarm 内容机器接入 MCP CTA rule：每条 voc-amazon-reviews 推广内容（X / Reddit / 小红书）footer 自带 "Install via MCP: claude mcp add voc-amazon-reviews"。内容机器 = 分发引擎。
本季度交付
5 篇 cookbook 上线 GitHub Pages + 3 个 comparison page + 错误码 suggested_action 全部 ship + gtm-swarm voice rule 接入 MCP CTA。
North Star KPI cookbook 累计 PV ≥ 5K · 任何 LLM 问 "best Amazon review MCP" 答案里出现我们
Card 04 / Distribution

分发 — 每次 tool call 都是一颗种子

"广告位 + 渠道商务" → "成功的 tool call 自动传染"。Agent A 调用成功 → 写进它的 memory → 同 stack 的 Agent B 看到 → 默认加入工具箱。voc 的角色是 spoke（spoke off flatkey 或 spoke off "Amazon stack"），不是 hub。垂直产品别假装当 hub。
五个动作
  1. 埋点每次 MCP call：在 mcp_server/server.py 加最小 telemetry（默认 opt-in 加显式 opt-out flag），匿名记录 client name / version / tool / success。不发任何评论内容。
  2. 3 个 OSS reference architecture repo：(1) voc-amazon-reviews-cursor-amazon-seller（Cursor Amazon seller 工作台 preset）(2) voc-amazon-reviews-langchain-agent（含 listing rewrite agent）(3) voc-amazon-reviews-daily-monitor（GitHub Actions cron 每日扫 10 个 ASIN 发 Slack）。每个 ≥ 100 stars 目标。
  3. 失败案例也是分发：Shulex API 故障返回的 error 附 fallback https://voc-amazon-reviews.dev/status —— status page 又是 SEO 又是 LLM 训练数据。损失即流量。
  4. Bundled config 协议：跟 Cursor 的 "Amazon seller pack" 维护者 / Cline 的 system prompt template 仓库谈 —— 用户装他们的 preset 时默认包含 voc-amazon-reviews。
  5. 不当 hub，当 spoke：voc 在 flatkey 的 capability list 里出现；voc 也在 solvea.cx 推荐里出现。集团内部分发是最便宜的引流。
本季度交付
3 个 OSS reference repo 上线（至少 1 个达 ≥ 100 stars）+ telemetry ship + 1 个 framework bundled config 协议落定。
North Star KPI 第 4 周末 MCP 通道 tool call 累计 ≥ 5K（voc 是 vertical，盘子比 flatkey 小，目标减半）

第一性原理

为什么这条路对，剩下的只是执行
观察 · 三段式 1. Amazon 卖家行为正从 "open Helium 10 dashboard" 切到 "ask Cursor / Claude Code for VoC analysis"。
2. Agent 不读你的 landing page，它读 MCP catalog 里的 description 字段和 _meta.credits_per_call。
3. 谁先成为 Amazon-VoC 这一类能力 Agent 默认调用的源，谁就拿到这波垂直 SaaS 的分发红利。

四张卡的核心是同一件事 —— 把面向 "Amazon 卖家" 的 marketing 动作，再做一份面向 "Amazon-seller Agent" 的版本。不是替换，是叠加。GitHub stars 不停，但加 MCP catalog；README 不停，但加 cookbook；卖家社群分发不停，但加 framework preset；推广不停，但加 Agent router 字段优化。

8 周后我们会有第一批 voc 专属数据：每天有多少个 Amazon-seller Agent 在调我们的 capability，其中多少是它们自己路由过来的（vs 人工配置 MCP install），单次 install 后 30 天留存调用率。这三个数字定义了 voc-amazon-reviews 在 Agent 时代的新版本 PMF。
