六个产品落地，锻造一个 Skill

agent-gtm 是一个 skill —— 把任意产品翻译成「发现 → 引流 → 内容 → 分发」四张卡的 Agent 时代 GTM 打法。把它写完整再用 → 用六个真实产品各跑一周,每次落地都把 skill 磨得更利。产品是试验场,skill 才是要交付的东西。

方法：4-card 打法 × 1 周冲刺 试验场：6 产品（1 hub + 5 spoke） 产物：被实战验证过的 agent-gtm skill 模板：每个产品 tab = btcmind playbook 完整结构
PRINCIPLES.md · §06 Agents 是新的分发渠道 —— 量级等同于 App Store / SEO / 公众号 / TikTok 各自的拐点。 Agent 不调用你的 UI,它调用你的能力。 只有界面没有 API 的产品会被绕过；API 不够 Agent-friendly 的,输给更 friendly 的那个。 谁先成为某类能力的默认调用源,谁就拿到这波分发红利。

Tool call & 默认工具箱
Card 01

发现

用户搜到你 → Agent 工具索引里看到你。索引来自 MCP 目录、LLM 训练数据、框架默认配置。

Card 02

引流

广告买点击 → Agent router 决定调谁。靠 tool description、cost 元数据、一行安装的相对优势胜出。

Card 03

内容

为人写 blog → 为 Agent 写 cookbook。OpenAPI、可跑示例、语义化错误码 ＞ 散文。

Card 04

分发

渠道 BD → 成功 tool call 自动传染。OSS 参考架构、框架 preset、hub-spoke 互链。

架构 · Hub-and-Spoke

flatkey 横向 hub（LLM 基础设施）· 5 个 vertical spoke 各占一个领域 · capability manifest 双向互链

每个 spoke 的 /.well-known/agent-capabilities.json 里列 flatkey 当 relation:"hub"； flatkey 的 related_capabilities 反向推荐 5 个 spoke。Agent router 探到任意节点,就能顺 manifest 走完整张图 —— 横向 hub × 5 个 vertical spoke = 一个默认配置候选包。集团内部互链是最便宜的流量。

为什么是「锻造 skill」而不是「做 GTM」

六次落地 = 六次对 skill 的压力测试 · 每次把缺口回写进 skill
STEP 1
选产品

6 个真实产品,覆盖 hub + 4 个 vertical 领域,差异越大越能暴露盲点。

STEP 2
跑四张卡

用 agent-gtm skill 当场生成 4-card + 1 周冲刺,不手工绕过。

STEP 3
落地执行

一周内真正 ship:catalog、manifest、cookbook、cross-link。

STEP 4
暴露缺口

哪一步 skill 给不出动作 / 分类错 / 模板不通用 —— 记下来。

STEP 5
回写 skill

缺口变成 references/ 规则、examples/ 案例。skill 变利,下个产品更快。

btcmind 这一轮已回写三条:(1) 垂直产品 catalog 分类不能跟 hub 抢 "AI/Inference",要进领域分类； (2) 语义化错误码不只是文档,docs/errors.md 必须配引擎里真实的 type+suggested_action； (3) cross-link manifest 要双向落代码,不是 PPT 概念。 skill 已开源:github.com/mguozhen/agent-gtm。六个产品跑完,agent-gtm 就是被六种业态验证过的 skill。
